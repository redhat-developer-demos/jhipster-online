Git provider (runtime)

Overrides stored in the database when OAuth is not provided via application properties. Client secrets are never shown; set a new value only when you want to replace the current secret.

OAuth setup (external docs)

Register the authorization callback URL on each provider exactly as your users reach this server (HTTPS), for example https://<your-jhipster-online-host>/api/github/callback and https://<your-jhipster-online-host>/api/gitlab/callback (Gitea: https://<your-jhipster-online-host>/api/gitea/callback). The GitLab Redirect URI field below must match what you register on GitLab.

GitHub
  • Creating a GitHub OAuth App · OAuth scopes · Developer Settings (github.com)
  • Scopes requested by JHipster Online (authorization link uses): user (profile), repo (create/push repositories), workflow (update GitHub Actions workflow files). These are the effective minimum for generators and CI features that touch repos and workflows.
GitLab
  • GitLab OAuth 2.0 provider / user-owned applications · Applications (gitlab.com)
  • Create a confidential application. Enable at least read_user and api (GitLab Java API for projects and merge requests). If your policy forbids full api, use your admin guide to grant the narrowest scopes that still allow repository creation and merge requests from the API.
Gitea
  • Gitea OAuth2 provider — register the same callback pattern under /api/gitea/callback; scopes depend on your Gitea version (typically read:user and repository write scopes for generating apps).
